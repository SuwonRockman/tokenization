# 토크나이제이션: 텍스트를 LLM이 이해하는 언어로 변환하기

> **핵심 질문:** LLM은 텍스트가 아닌 숫자만 이해합니다. 그렇다면 "machine learning"이라는 단어를 어떻게 숫자로 변환할까요? 그리고 "machines", "learning", "learnings" 같은 변형된 단어들은?

---

## 📌 배경: 왜 토크나이제이션이 어려운가?

### 문제의 핵심

LLM이 텍스트를 처리하려면 **텍스트를 작은 단위로 나누고 숫자로 변환**해야 합니다. 하지만 단순하지 않습니다:

```
사람의 관점:              LLM의 관점:
"machine" → 의미 직관적    "machine" → 어떻게 나눌 것인가?

선택지 1: 문자 단위
"machine" → ['m','a','c','h','i','n','e'] → [1,2,3,4,5,6,7]
→ 7개 토큰 (정보 분산, 비효율)

선택지 2: 단어 단위
"machine" → ['machine']
"machines" → ['machines']  ← 새로운 ID 필요!
→ 어휘 폭발 (메모리 낭비)

선택지 3: 적절한 크기의 단위? ← 이것이 토크나이제이션의 목표
```

### 실제 영향

| 언어 | 문자 단위 | 단어 단위 | 토크나이제이션 | 비용 차이 |
|------|----------|----------|--------------|---------|
| 영어 | "hello world" → 11개 | 2개 | 2개 | 1배 |
| 한글 | "안녕하세요" → 5개 | 불명확 | 2개 | 2.5배 |
| 중국어 | 불가능 | 불명확 | 3개 | 3배+ |

**결론:** 토크나이제이션 선택에 따라 **처리 비용이 3배까지 차이**남!

---

## 🔍 핵심 개념: 적절한 크기의 단위 찾기

### 문제 정의

텍스트를 나누는 세 가지 극단:

```
← 너무 작음              최적                  너무 큼 →
문자 단위         서브워드 토크나이제이션      단어 단위

'h','e','l','l','o' → ['hel','lo'] → ['hello']
      11개              2개         1개

문제:             최적:           문제:
정보분산        의미+효율         어휘폭발
비효율          균형잡힘          OOV많음
```

### 토크나이제이션의 목표

**"텍스트의 의미를 최대한 보존하면서, 가장 효율적인 크기의 단위로 나누기"**

---

## 💡 3가지 솔루션과 기술적 차이

### 1️⃣ BPE (Byte Pair Encoding) - 빈도 기반

**원리:** 데이터에서 가장 자주 나오는 문자/바이트 쌍을 반복 합침

```
Step 1: ['h', 'e', 'l', 'l', 'o']
        ('l','l')이 가장 자주 나옴 → 'll'로 치환
Step 2: ['h', 'e', 'll', 'o']
        ('e','ll')이 자주 나옴 → 'ell'로 치환
Step 3: ['h', 'ell', 'o']
```

**사용 모델:** GPT-2, GPT-3, GPT-4, Llama

**기술적 특징:**
- Greedy 알고리즘 (항상 가장 자주 나오는 패턴 우선)
- 의미 무시 (빈도만 고려)
- 공백 표시: Ġ 사용
- 어휘 크기: 약 50K개

**한글 처리 문제:**
```
"기계 학습을 좋아합니다"
↓ (BPE 처리)
['기', '계', '학', '습', '을', '좋', '아', '합', '니', '다']  ← 10개!
→ 자음/모음 분리로 의미 완전 손실
```

---

### 2️⃣ WordPiece - 확률 기반

**원리:** 미리 정의된 어휘 사전에서 확률이 높은 단어 조합 우선 선택

**사용 모델:** BERT, RoBERTa, DistilBERT, KoBERT

**기술적 특징:**
- 사전 기반 (학습 시점에 어휘 고정)
- ##마크로 서브워드 연결 명시
- 확률 기반 (발생 확률 최대화)
- 어휘 크기: 약 30K개

**한글 처리:**
```
"학습을 합니다"
↓ (띄어쓰기 기준)
['학습', '##을', '합니다']  ← 완벽하지 않음
```

---

### 3️⃣ SentencePiece - 언어 독립적

**원리:** 모든 언어를 동일하게 바이트 레벨에서 처리

**사용 모델:** mBART, mT5, XLM-RoBERTa, Llama 2

**기술적 특징:**
- 언어 무관 (특정 언어 의존성 없음)
- 공백 표시: ▁ 사용
- 바이트 기반 (가장 기본 단위에서 시작)
- 어휘 크기: 약 32K~64K개

**한글 처리 최적:**
```
"기계 학습을 좋아합니다"
↓ (SentencePiece)
['▁기계', '▁학습', '을', '▁좋아합니다']  ← 5개
→ 의미 보존 + 효율적 + 자음/모음 분리 없음
```

## 📊 실제 영향 분석

### 1. 토큰 수 차이 (비용에 직결)

```
같은 의미의 한글 문장:
"기계 학습을 좋아합니다"

BPE (GPT-2):        10개 토큰 → $0.005  (API 비용)
WordPiece (KoBERT):  6개 토큰 → $0.003
SentencePiece (XLM): 5개 토큰 → $0.0025

→ BPE는 SentencePiece의 4배 비쌈!
```

### 2. 다국어 서비스의 공정성 문제

```
BPE/WordPiece (영어 중심):
"Hello" → 1개 토큰
"안녕하세요" → 5개 토큰
→ 언어별 처리 비용 편차 5배!

SentencePiece (모든 언어 동등):
"Hello" → 1개 토큰
"안녕하세요" → 3개 토큰
→ 언어별 편차 최소화
```

### 3. 성능 영향

```
Attention 계산량 = O(n²)

토큰 수 2배 증가 → 계산량 4배 증가
메모리 2배 증가 → 배치 사이즈 4분의 1 감소
```
